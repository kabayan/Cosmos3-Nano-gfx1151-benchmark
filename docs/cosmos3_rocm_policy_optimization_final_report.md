## 2. 実施した主要な最適化施策

### ① モンキーパッチによる Graph Break (ホスト・デバイス同期) の全解消
*   **課題**: トランスフォーマーの各アテンション層の内部（`factored_from_joint_sequence` 内）で呼び出されていた `.tolist()` チェックが CPU-GPU 同期を引き起こし、グラフが細切れに切断されてコンパイル効率が著しく低下していました。
*   **対策**: アサーションロジックをバイパスし、配列演算をテンソル内で閉じるモンキーパッチを [run_cosmos_framework_policy_rocm.py](file:///home/kabayan/workspace/cosmos3/scripts/run_cosmos_framework_policy_rocm.py) に実装しました。
*   **効果**: グラフ全体の統一的な `torch.compile` が可能となり、1ステップあたりの実行時間が 1.99s/it ➔ 1.15s/it まで大幅に高速化されました。

### ② サンプリングループ内の微細な CPU-GPU 同期の排除
*   **課題**: 30ステップのサンプリングループの内部（`_get_velocity`）において、毎ステップで条件マスクが 0 超かどうかを評価する `noisy_mask.sum() > 0` が実行され、都度 CPU-GPU 同期が発生していました。
*   **対策**: 条件分岐を完全に排除し、常にテンソル演算として `pred * noisy_mask` を実行する最適化パッチ（`patched_get_velocity`）を適用しました。
*   **効果**: ループ内の同期が 0 回となり、CUDA Graphs (HIP Graphs) による一気通貫のキャプチャが可能な構造へと平坦化されました。

### ③ CUDA Graphs (HIP Graphs) の有効化
*   **対策**: 同期が排除された状態で `cpp_wrapper = False` および `triton.cudagraphs = True` を指定し、Inductor が生成する Python 実行パスで CUDA Graphs を有効化しました。
*   **効果**: 第1ステップで発生していた再コンパイル遅延（cache miss）が解消し、CPU-GPU 間のコマンド送信オーバーヘッドが極小化されました。ステップ平均速度が **1.145s/it ➔ 1.127s/it** へとさらに短縮されました。

### ④ PyTorch TunableOp による GEMM カーネルの最適化
*   **対策**: `PYTORCH_TUNABLEOP_TUNING=1` を用いて、Policy Model 実行時に使われる動的な行列乗算（GEMM）形状をスキャンし、AMD GPU（gfx1151）で最も実行速度の速い Tensile/hipBLASLt のカスタムカーネルを自動選択させました。
*   **効果**: チューニングテーブル（`tunableop_results00.csv`）が更新され、デフォルト実装よりも高速な演算器が最適に配備されました。

## 3. 結論と今後の展開
本最適化により、NVIDIA Cosmos3-Nano Policy Model を AMD ROCm 環境下で稼働させるにあたり、**総実行時間 41.66秒** という極めて優れた性能水準を達成し、対論文比 1.5倍以内の目標をクリアしました。

現在、同期処理としてのポテンシャルは限界まで引き出されており、さらなる実用的な速度向上を目指す場合は、当初の制約を緩和した以下の施策が考えられます：
1.  **非同期ビデオデコードによる制御遅延の極小化**:
    ロボットの制御側に対してアクションテンソル（アクションのみであれば数ミリ秒でデコード完了）をサンプリング終了（約33.8秒）の直後に即時返却し、最も重い VAE ビデオデコード（7.5秒）は別スレッドで非同期に処理する（アクション応答時間：**実質33.8秒**）。
2.  **生成解像度やサンプリングステップ数の調整**:
    サンプリングステップ数を 20 steps に減らす、あるいは生成解像度を下げることで、クオリティとトレードオフにしつつ実行時間を 20秒台前半まで短縮する。
